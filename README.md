# Sistema-Distribuido-en-3-Capas-Gaona-Axel
Guía de Implementación: Arquitectura Distribuida en 3 Niveles (Ubuntu Server & VirtualBox)Este repositorio contiene la documentación y el código para el despliegue de una infraestructura distribuida real. El proyecto se basa en la segmentación de funciones en tres entornos virtuales independientes utilizando Linux, con el objetivo de aislar la persistencia de datos, el procesamiento de la lógica de negocio y la interfaz de usuario final.📐 Diseño de la Infraestructura de RedPara la comunicación entre nodos, se configuró una Red NAT privada (10.0.2.0/24) con direccionamiento IP estático. Esto permite que los servidores se vean entre sí de forma interna y mantengan acceso a la red externa para la gestión de paquetes.ServidorFunciónDirección IP (Interna)Puerto de AplicaciónPuerto SSH (Host)VM1-DBAlmacenamiento de Datos10.0.2.153306 (MariaDB)2221VM2-BACKENDProcesamiento y API10.0.2.163454 (Node.js)2222VM3-FRONTENDPresentación y Web10.0.2.178080 (Apache2)2223🛠️ Stack TecnológicoVirtualización: Oracle VirtualBox con Ubuntu Server 24.04 LTS.Base de Datos (Capa 1): Motor relacional MariaDB.Servicios Backend (Capa 2): Node.js con Express, utilizando controladores MySQL2 y soporte CORS.Interfaz de Usuario (Capa 3): Web Server Apache2, maquetado en HTML5 y consumo de servicios mediante JavaScript (Fetch API).Herramientas de Desarrollo: VS Code, protocolos SSH y SCP para transferencia de archivos.Proceso de Despliegue1. Configuración del Entorno de RedEn VirtualBox, se debe crear una Red NAT (denominada NatNetwork) con el segmento 10.0.2.0/24. Es fundamental que el DHCP esté activo inicialmente para facilitar la instalación de los componentes base.2. Preparación de la Plantilla Maestra (ubuntu-base)Se instaló una instancia base de Ubuntu Server con 1.5 GB de RAM y 2 núcleos de CPU. Durante este proceso:Se habilitó el servicio openssh-server para gestión remota.Se optimizó el arranque del sistema desactivando los servicios de cloud-init y las esperas de red innecesarias para acelerar el inicio de cada nodo clonado.3. Clonación y Red Estática (Netplan)Se generaron tres clones (VM1, VM2, VM3) a partir de la imagen base, asegurando la reinicialización de las direcciones MAC. En cada nodo, se procedió a fijar la IP mediante la edición del archivo de configuración en /etc/netplan/:Fragmento de código# Ejemplo de configuración estática para los nodos
network:
  version: 2
  renderer: networkd
  ethernets:
    enp0s3:
      dhcp4: false
      addresses: [10.0.2.X/24] # Reemplazar X según el nodo
      routes:
        - to: default
          via: 10.0.2.1
      nameservers:
        addresses: [8.8.8.8, 8.8.4.4]
4. Mapeo de Puertos (Port Forwarding)Para administrar el sistema desde la máquina anfitriona, se establecieron reglas de reenvío en la configuración de la Red NAT de VirtualBox, vinculando los puertos locales (127.0.0.1) con las IPs correspondientes de las máquinas virtuales.A partir de aquí, el acceso se realiza mediante SSH:Bashssh axelon@localhost -p 2221  # Acceso a la VM1
Configuración Específica por CapaVM1: Gestión de Datos (MariaDB)Se instaló el motor de base de datos y se creó el esquema escuela. Se configuró un usuario específico (usuario_consulta) con permisos remotos.Punto clave: Se modificó el archivo 50-server.cnf estableciendo bind-address = 0.0.0.0 para permitir que el Backend (VM2) se conecte a la base de datos.VM2: Lógica de Negocio (Node.js)El servidor utiliza Express para exponer dos puntos de acceso (endpoints):POST /grabaAlumnos: Recibe datos, valida que el DNI no esté repetido e inserta el registro.GET /consultarAlumnos: Recupera la lista de alumnos ordenados alfabéticamente.Para preparar el entorno:Bashssh axelon@localhost -p 2222
